# AIGC 审核工作流（MVP）技术架构与火山引擎选型方案

## 1. 目标与约束

### 1.1 业务目标（与现有需求对齐）

- 支撑 4 个审核节点（资产、视觉、视听整合、成片）闭环流转。
- 支撑“人工审核 + 自动回炉重生成 + 版本追溯”核心能力。
- 满足多人协作：任务优先级、领取锁、超时释放、并发读写隔离。
- 支撑角色隔离：质检员、剪辑中台、合作方（脱敏访问）。

### 1.2 技术约束

- 云环境：优先采用火山引擎托管服务，减少自建运维面。
- 数据库：已落地 PostgreSQL schema，需优先兼容。
- 架构策略：MVP 快速上线、后续平滑扩展到多项目并行生产。

---

## 2. 总体架构设计（推荐）

采用“前后端分离 + 事件驱动编排 + 托管数据库/缓存/对象存储”的分层架构：

1. **接入层**
   - Web 前端（审核台）通过 HTTPS 访问。
   - 统一入口：负载均衡 + API 网关（鉴权、限流、审计透传）。
2. **应用层**
   - 核心业务服务（任务面板、节点审核、版本与归因）。
   - 工作流编排服务（回炉分发、状态推进、异步回调处理）。
3. **数据与媒体层**
   - PostgreSQL（事务数据）、Redis（锁与热点缓存）、消息队列（异步任务）、对象存储（音视频与候选素材）。
4. **AI 与外部集成层**
   - 对接既有 Agent/模型接口（生成、修订摘要、归因判定）。
5. **治理层**
   - 统一日志、指标、链路追踪、审计与告警。

## 3. 火山引擎云服务选型（主方案）

| 架构域 | 推荐服务 | 作用 | 选型原因 |
|---|---|---|---|
| 网络隔离 | VPC + 子网 + 安全组 + NAT 网关 | 内外网隔离、最小暴露面 | 基础必选，满足生产分区 |
| 外部入口 | CLB/ALB + WAF（可选） | 统一 HTTPS 入口、防护 | 简化公网暴露与攻击防护 |
| API 统一治理 | API 网关（APIG） | 鉴权、限流、灰度、版本管理 | 减少网关自建复杂度 |
| 计算平台 | VKE（托管 Kubernetes） | 承载后端微服务与异步消费者 | 与后续弹性扩容、服务拆分兼容 |
| 镜像仓库 | 容器镜像仓库 CR | 镜像托管与分发 | 与 VKE 原生协同 |
| 关系型数据库 | RDS for PostgreSQL（高可用版） | 事务主库 | 兼容现有 schema，托管备份恢复 |
| 缓存 | Redis（托管版） | 任务锁、短期状态缓存、热点查询 | 低延迟，减少 DB 竞争 |
| 消息队列 | Kafka/RocketMQ（托管版） | 回炉、重生成、回调异步解耦 | 解耦外部接口抖动与峰值流量 |
| 对象存储 | TOS | 视频/关键帧/音频/成片文件存储 | 成本低、可配 CDN、生命周期管理 |
| 日志与检索 | TLS（日志服务） | 结构化日志、审计检索 | 支撑排障与合规留存 |
| 监控告警 | 云监控（Cloud Monitor） | 指标、告警、SLO 监控 | 托管告警闭环 |
| 密钥管理 | KMS + Secrets Manager（如已开通） | 密钥与凭据托管 | 降低明文泄露风险 |
| CI/CD | 火山引擎流水线（或 GitHub Actions + CR/VKE） | 自动构建部署 | 支撑环境一致性与灰度发布 |

> 备注：具体商品名在控制台可能因版本升级有差异，落地时以火山引擎当前标准商品目录为准。

## 4. 关键方案选型对比与结论

### 4.1 计算形态：VKE vs ECS 直部署

- **推荐：VKE**
  - 优点：弹性扩缩容、服务治理友好、后续多服务演进成本低。
  - 成本：早期平台复杂度略高于单机 ECS。
- 备选：ECS + Docker Compose（仅 PoC）
  - 适用于 2 周内验证，不建议作为 MVP 生产方案。

### 4.2 数据库：RDS PostgreSQL vs 自建 PostgreSQL

- **推荐：RDS PostgreSQL 高可用版**
  - 优点：自动备份、监控、故障切换、运维风险低。
- 备选：ECS 自建 PG
  - 仅在强定制插件且托管版无法满足时考虑。

### 4.3 异步总线：Kafka vs RocketMQ

- **推荐：Kafka（若团队已有流处理经验）**
  - 适合高吞吐事件流（回炉、状态变更、埋点）。
- 备选：RocketMQ（若团队偏事务消息语义）
  - 对业务命令型消息也较友好。

### 4.4 鉴权：网关统一鉴权 vs 应用内分散鉴权

- **推荐：网关做统一入口鉴权 + 应用做细粒度 RBAC 二次校验**
  - 保证“角色可见性 + 操作权限”双保险。

## 5. 业务到技术的核心链路设计

### 5.1 任务面板与领取锁（US-1~US-5, US-28）

- `stage_tasks` 为主事实表，`priority_group + deadline + created_at` 组合排序。
- 领取流程：
  1. API 网关鉴权通过后调用任务服务。
  2. 任务服务先查 Redis 锁（快速失败）再落库事务更新 `locked_by/lock_expire_at`。
  3. 成功后发布 `task.claimed` 事件到 MQ。
- 超时释放：
  - 定时任务扫描 `lock_expire_at`，释放锁并发出 `task.released` 事件。

### 5.2 节点审核与打回归因（US-24, US-31）

- 审核动作写入 `review_points/review_decision_records/feedback_records`。
- 打回后发布 `review.rejected` 事件，由编排服务处理：
  - 归因阶段判定（人工指定优先，规则引擎兜底）。
  - 创建目标节点 `stage_tasks`（状态 `generating/pending`）。
  - 调用外部 Agent 生成接口并等待回调。

### 5.3 版本追溯与媒体产物（US-29）

- 每次回炉生成新 `episode_versions` 记录，旧版本只读。
- 媒体文件写入 TOS（建议路径：`project/series/episode/version/stage/...`）。
- `revision_logs` 记录修订摘要，供节点4 左侧历史展示。

### 5.4 角色隔离与脱敏（US-27）

- 合作方访问单独 API 路由域（或网关路由策略）：
  - 仅返回节点4必要字段。
  - 屏蔽资产内部标识、模型参数、内部评分元数据。

## 6. 数据架构设计

### 6.1 事务数据层

- 主库：RDS PostgreSQL（多可用区高可用）。
- 读写策略：MVP 先单写实例，后续按读流量增加只读实例。
- 一致性：
  - 关键流程（领取锁、通过/打回、版本推进）必须走数据库事务。
  - MQ 采用 Outbox 模式避免“写库成功但消息丢失”。

### 6.2 缓存与锁

- Redis 仅做加速与分布式锁辅助，不作为最终真相源。
- 锁双写策略：Redis 锁 + DB 条件更新，确保幂等与可恢复。

### 6.3 对象存储与分发

- TOS 存储所有素材与成片，开启生命周期分层：
  - 热数据（近 30 天）标准存储。
  - 冷数据（历史版本）低频/归档存储。
- 外部播放建议前接 CDN，减少跨区域加载延迟。

## 7. 安全与合规设计

- 网络：应用节点仅内网访问 DB/Redis/MQ；仅网关与负载均衡暴露公网。
- 身份：统一 JWT/OIDC（接企业 IdP），服务间鉴权走 mTLS（后续阶段）。
- 数据安全：
  - TOS、RDS、Redis 均启用静态加密（KMS 管理密钥）。
  - 传输全链路 TLS。
- 审计：
  - 关键操作写 `audit_logs` + TLS 日志双落盘。
  - 审计日志留存建议 >= 180 天（按合规要求调整）。

## 8. 可观测与运维体系

### 8.1 指标（必须）

- 业务指标：
  - 每节点通过率、驳回率、平均回炉轮次。
  - 任务领取到完成耗时 P50/P95。
- 技术指标：
  - API 延迟/错误率、DB 慢查询、队列积压长度、回调失败率。

### 8.2 日志与链路

- 日志规范：JSON 结构化，统一 `trace_id`, `episode_id`, `task_id`, `version_id`。
- 链路追踪：网关到应用到外部接口全链打点，便于定位“哪次回炉失败”。

### 8.3 告警基线

- P0：主流程不可用（任务无法领取、无法提交通过/驳回）。
- P1：队列积压超过阈值、回炉回调超时、DB 连接池耗尽。
- P2：单节点错误率升高、部分接口延迟异常。

---

## 9. 高可用与容灾

- 生产建议多可用区部署：
  - VKE 工作节点跨可用区。
  - RDS/Redis 采用高可用版。
- 备份策略：
  - RDS 自动备份 + 每日快照。
  - TOS 开启版本控制（关键路径）和跨区域复制（如有监管要求）。
- 容灾目标（建议）：
  - RPO <= 5 分钟，RTO <= 30 分钟（MVP 阶段）。

---

## 10. 容量与成本预估（MVP 级）

### 10.1 初始规模假设

- 日处理：100~500 集审核流转。
- 峰值并发：50~200 在线审核用户。
- 媒体增量：1~3 TB/月（取决于回炉轮次与分辨率）。

### 10.2 初始资源建议

- VKE：
  - 业务服务 3~6 个副本（按接口 QPS 自动扩缩）。
  - 异步消费者 2~4 个副本（按队列积压扩缩）。
- RDS PostgreSQL：
  - 4C16G 起步，高可用，100~300GB SSD。
- Redis：
  - 2C4G 起步，开启持久化与主从高可用。
- MQ：
  - 3 broker 起步（托管版规格按吞吐选）。
- TOS：
  - 标准存储 + 生命周期规则（30/90/180 天分级）。

## 11. 分阶段落地路线图

### Phase A（2~4 周）：MVP 可用

- 完成 T1~T4（数据底座、任务面板、锁机制、节点1）。
- 云上只启用核心：VKE + RDS + Redis + TOS + TLS。

### Phase B（4~8 周）：完整流程

- 补齐节点2/3/4、归因回炉、版本回放。
- 引入 MQ 解耦外部生成调用与回调处理。

### Phase C（8~12 周）：生产增强

- 多可用区容灾、网关限流、细粒度告警、成本优化。
- 加入 CDN、冷热分层、只读实例、灰度发布。

---

## 12. 关键风险与规避策略

- 外部 Agent 接口不稳定：
  - 通过 MQ + 重试 + 死信队列 + 幂等键降低故障扩散。
- 任务锁冲突导致用户体验差：
  - Redis 快速锁 + DB 条件更新 + 锁可视化提示。
- 媒体成本增长过快：
  - 版本清理策略、冷热分层、只保留关键中间产物。
- 合作方数据越权：
  - 独立视图 DTO + 网关策略 + 审计追踪。

---

## 13. 最终推荐结论

在火山引擎上，建议以 **VKE + RDS PostgreSQL + Redis + MQ + TOS + APIG + TLS/云监控** 为核心组合，形成“事务一致性 + 异步解耦 + 媒体分层存储 + 可观测闭环”的架构底座。

该方案与现有 `requirements/design/data-structures` 可直接对齐，能在不推翻当前数据模型的前提下，支撑 MVP 快速上线，并保留后续向多项目、多人并发、跨角色协作的扩展空间。
