# 火山引擎服务开通待办（主线架构）

本文档用于收敛当前主线架构下的云服务开通顺序，避免一次性铺太多服务导致成本和运维复杂度上升。原则是：先补齐 MVP 运行面，再补自动化与治理，最后补正式公网入口与对外分发。

## 1. 当前已开通

- [x] `VPC / 子网 / 安全组`
- [x] `ECS` 跳板机
- [x] `RDS PostgreSQL`
- [x] `Redis`
- [x] `VKE`
- [x] `CR`
- [x] `TOS`

## 2. 必须马上开通

这些项不依赖域名备案，建议优先执行。

### 2.1 日志与排障

- [ ] 开通 `TLS` 日志服务
- [ ] 创建项目日志集与日志主题，至少区分：
  - 应用日志
  - 审核工作流日志
  - VKE 容器运行日志
- [ ] 明确日志保留周期（建议开发期 7~15 天）

### 2.2 监控与告警

- [ ] 开通云监控 / 告警能力
- [ ] 为 `VKE` 配置基础监控项：
  - 节点 CPU / 内存 / 磁盘
  - Pod 重启次数
  - Pod Pending / CrashLoopBackOff
- [ ] 为 `RDS` 配置基础告警：
  - 连接数异常
  - CPU 异常
  - 存储空间不足
- [ ] 为 `Redis` 配置基础告警：
  - CPU / 内存使用率
  - 连接数异常
  - 慢查询或阻塞风险

### 2.3 部署自动化

- [ ] 选定 `CI/CD` 方案：
  - 火山引擎流水线
  - 或 `GitHub Actions + CR + VKE`
- [ ] 明确部署链路：
  - 代码拉取
  - 镜像构建
  - 推送 `CR`
  - 部署到 `VKE`
- [ ] 准备最小权限发布账号，而不是长期使用高权限 AK/SK

## 4. 受域名与备案影响的项

这些项建议在域名准备完成后推进，否则现在只能先用公网 IP 或测试入口。

### 4.1 正式公网入口

- [ ] 购买或迁入正式域名

- [ ] 完成 ICP 备案
- [ ] 为 `VKE / Ingress / LB` 绑定正式域名
- [ ] 申请并配置 HTTPS 证书

### 4.2 对外接入增强

- [ ] 按需开通 `CDN`
- [ ] 按需开通 `API 网关`

## 5. 当前建议执行顺序

1. `TLS`
2. 云监控 / 告警
3. `CI/CD`
4. `KMS / Secrets Manager`
5. 消息队列
6. 域名 + ICP 备案 + HTTPS + 正式入口
7. `CDN / API 网关 / WAF`

## 6. 域名与备案说明

### 6.1 现在没有域名会影响什么

- 不影响当前内网开发、跳板机访问、数据库/缓存/对象存储接入
- 不影响先用公网 `LB IP` 做临时联调
- 会影响正式对外 Web 入口、稳定 HTTPS 访问、合作方长期访问入口

### 6.2 在国内通常需要你做什么

- 准备一个正式域名
- 完成域名实名认证
- 以企业主体发起 ICP 备案
- 准备备案材料（通常包括企业信息、负责人信息、联系方式等）
- 备案通过后，再做 DNS 解析和 HTTPS 证书配置

### 6.3 当前阶段的建议

- 先推进不依赖域名的基础设施项
- 域名备案并行推进，不阻塞当前 MVP 内测和内网联调
- 在正式开放给合作方或客户前，再切到“域名 + HTTPS + 正式入口”方案

## 7. 执行备注

- 当前阶段默认先开通不依赖域名的能力
- 任何会增加长期成本或对公网暴露面的服务，在正式上线前再做最终收口
