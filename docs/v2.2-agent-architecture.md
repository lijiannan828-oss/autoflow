## 2. 三层决策模型（Three-Layer Decision Model）

> 来源：§2.4 "Agent三层决策模型——策划、执行、复盘"

Agent 不再按"每镜头一次完整 LLM 推理"运行，而是采用**集级策划→镜头级执行→批后复盘**三层模型，
将每集 LLM 调用从 60-120 次降至 3-5 次。

### 2.1 三层概览

| 层级 | 入口方法 | 触发节点 | LLM 调用 | 职责 |
|------|---------|---------|---------|------|
| **第一层：集级策划** | `plan_episode()` | N02/N06 | 1 次（覆盖全集 30-60 镜头） | 产出"全集作战计划"：镜头参数表、资源分配、风格锚定 |
| **第二层：镜头级执行** | `execute_shot(plan)` | N10/N14 | N10: Phase1 1次LLM/shot（prompt编排）+ Phase2 零LLM; N14: 零LLM | 按计划逐镜执行，N10 Phase1 为每帧×每候选生成独立 prompt，confidence 低于阈值时降级回 LLM 单次推理 |
| **第三层：批后复盘** | `review_batch(results)` | N12/N16 | 1 次（多模态分析全集产物连续性） | 检查全集一致性、沉淀经验教训到记忆 |

### 2.2 LLM 调用对比

| 维度 | 旧（每镜头决策） | 新（三层模型） |
|------|---------------|--------------|
| 每集 LLM 调用 | 60-120 次（每镜头完整推理） | 3-5 次决策 + N10 每镜头 1 次 prompt 编排（轻量） |
| 单集决策成本 | ￥3-6 | 决策 ￥0.15-0.3 + N10 prompt ￥0.3-0.6 |
| 延迟 | 每镜头 2-5s LLM 等待 | 集级 plan 10-15s + N10 Phase1 ~2-5s/shot(可并行) + Phase2 纯 GPU |

### 2.3 BaseAgent 入口

`BaseAgent.execute()` 保持统一入口，内部按 `mode` 分流：

```python
class BaseAgent(ABC):
    def execute(self, context: AgentContext) -> AgentResult:
        """统一入口，按 context.mode 分流三条路径。"""
        if context.mode == "plan":
            return self._run_plan(context)      # 第一层
        elif context.mode == "shot":
            return self._run_shot(context)       # 第二层
        elif context.mode == "review":
            return self._run_review(context)     # 第三层
        else:
            return self._run_legacy(context)     # 兼容旧四步循环
```

- `_run_plan`：recall → plan_episode（1 次 LLM）→ reflect
- `_run_shot`：execute_shot（N10: Phase1 1次LLM prompt编排 + Phase2 零LLM; 其他节点: 零LLM 纯规则+记忆快查）；confidence < 阈值时 fallback 到单次 LLM
- `_run_review`：review_batch（1 次多模态 LLM）→ reflect（沉淀经验）
- `_run_legacy`：recall → reason → act → reflect（向后兼容，Supervisor/EvolutionEngine 等非生产 Agent 使用）

### 2.4 各 Agent 在三层中的角色

> **核心原则**：不是每个 Agent 都实现全部三层。每个 Agent 只实现与其职责匹配的层级，
> 未实现的方法保持基类默认（raise NotImplementedError），workers.py 通过 NODE_DECISION_LAYER
> 确保只调用该 Agent 已实现的层。

| Agent | plan_episode | execute_shot | review_batch | 需实现的层 | 说明 |
|-------|:-:|:-:|:-:|:-:|------|
| **ScriptAnalyst** | ✅ N01 | — | — | plan only | 剧本解析是一次性集级分析，无逐镜头执行 |
| **ShotDesigner** | ✅ N02 | ✅ N04/N05/N16b | ✅ N16 | 全三层 | 最典型的三层 Agent：全集分镜→逐镜填参→连续性复盘 |
| **VisualDirector** | ✅ N06 | ✅ N07/N09/N10/N13/N14/N17/N19 | — | plan + shot | 集级视觉策略 + 逐镜头生成；视觉连续性由 QualityInspector 负责 |
| **AudioDirector** | ✅ N07b | ✅ N20/N22 | — | plan + shot | 音色方案（plan）+ BGM/SFX/混音（shot）；音频质检由 QI 负责 |
| **Compositor** | — | ✅ N23/N25/N26 | — | shot only | 纯执行：按前序节点产物合成，不需要集级策划 |
| **QualityInspector** | — | — | ✅ N03/N11/N12/N15 | review only | 所有 QC 节点本质都是"批后复盘"——检查前序产物质量 |
| **ReviewDispatcher** | — | — | — | **legacy** | 解析人审批注→拆任务，用 reason()+act() 更自然 |
| Supervisor | — | — | — | **legacy** | 横切守卫，规则引擎+LLM，不走三层 |
| EvolutionEngine | — | — | — | **legacy** | 独立进化循环，不走三层 |

### 2.5 NODE_DECISION_LAYER 映射规范

> 此映射是 workers.py 设置 `context.mode` 的唯一依据。编排运行时 Agent 在 R6 中实现。

```python
# topology.py 或 workers.py 中定义
NODE_DECISION_LAYER: dict[str, DecisionMode] = {
    # ── 第一层：集级策划（plan_episode）──
    "N01": "plan",     # ScriptAnalyst: 剧本解析
    "N02": "plan",     # ShotDesigner: 全集分镜
    "N06": "plan",     # VisualDirector: 全集视觉策略
    "N07b": "plan",    # AudioDirector: 音色方案

    # ── 第二层：镜头级执行（execute_shot）──
    "N04": "shot",     # ShotDesigner: 逐镜头参数填充
    "N05": "shot",     # ShotDesigner: 镜头细化
    "N07": "shot",     # VisualDirector: 美术生成
    "N09": "shot",     # VisualDirector: 关键帧优化
    "N10": "shot",     # VisualDirector: 关键帧生成
    "N13": "shot",     # VisualDirector: 视频预处理
    "N14": "shot",     # VisualDirector: 视频生成
    "N16b": "shot",    # ShotDesigner: 影调节奏调整
    "N17": "shot",     # VisualDirector: 视频后处理
    "N19": "shot",     # VisualDirector: 字幕合成
    "N20": "shot",     # AudioDirector: BGM/SFX 生成
    "N22": "shot",     # AudioDirector: 音频混音
    "N23": "shot",     # Compositor: 视听合成
    "N25": "shot",     # Compositor: 片头片尾
    "N26": "shot",     # Compositor: 最终输出

    # ── 第三层：批后复盘（review_batch）──
    "N03": "review",   # QualityInspector: 分镜质检
    "N11": "review",   # QualityInspector: 关键帧质检
    "N12": "review",   # QualityInspector: 视觉连续性
    "N15": "review",   # QualityInspector: 视频质检
    "N16": "review",   # ShotDesigner: 连续性复盘

    # ── Legacy（不走三层）──
    "N08": "legacy",   # ReviewDispatcher: Gate
    "N18": "legacy",   # ReviewDispatcher: Gate
    "N21": "legacy",   # ReviewDispatcher: Gate
    "N24": "legacy",   # ReviewDispatcher: Gate
}
```

**规则**：
1. 每个 node_id 有且仅有一个 mode，不存在"同一节点跑两层"的情况
2. worker 工厂在创建 `AgentContext` 时从此映射读取 mode，设置 `context.mode`
3. 未在映射中的节点默认 `"legacy"`
4. Agent 只需实现上表中涉及的层级方法，其余保持基类默认

### 2.6 Trace 类型

`agent_traces.trace_type` 新增值：

| trace_type | 层级 | 说明 |
|-----------|------|------|
| `plan` | 第一层 | 集级策划输出（全集作战计划） |
| `execute` | 第二层 | 镜头执行参数+自检结果 |
| `execute_fallback` | 第二层 | confidence 不足时降级 LLM 推理 |
| `review` | 第三层 | 批后复盘分析结果 |
| `recall` | 兼容 | 旧循环的记忆检索 |
| `reason` | 兼容 | 旧循环的 LLM 推理 |
| `act` | 兼容 | 旧循环的执行 |
| `reflect` | 通用 | 经验沉淀（plan/review 后都会触发） |

所有 trace 自动记录到 `agent_traces` 表（trace_type/input_summary/output_summary/cost/duration）。

## 3. 三层记忆架构

```
Working Memory (PipelineState)     ← 单次执行上下文
    ↓
Project Memory (PG agent_memory)   ← 跨执行持久化
    ↓
Long-term Memory (Qdrant RAG)      ← 语义检索 + 案例库
```

### 3.1 Working Memory
- 存储：LangGraph `PipelineState` TypedDict
- 生命周期：单次流水线执行
- 用途：节点间数据传递

### 3.2 Project Memory
- 存储：`core_pipeline.agent_memory` 表
- 字段：agent_name, memory_type, scope, content_key, content_value, confidence, access_count
- CRUD：`backend/common/agent_memory.py`
- 清理策略：超过 `stale_days` 未访问 → confidence × decay_factor → 低于阈值则删除

### 3.3 Long-term Memory (RAG)
- 存储：Qdrant `autoflow_rag` 集合（dim=1536, COSINE）
- 记录：`RAGCase`（chain_id, quality_score, case_type, genre, scene_type, payload）
- 写入：EvolutionEngine 在反思模式中写入高分案例
- 检索：各 Agent 在 recall() 阶段按 genre/scene_type 过滤 + quality_score 排序
- 客户端：`backend/common/rag.py` → `get_rag_client()`

## 4. Prompt 资产架构

三层覆写：

```
Master Template (prompt_assets)
    ↓ genre_tag 匹配
Genre Adapter (genre_adapters)
    ↓ {{var}} 替换
Instance Adapter (运行时注入)
```

- Master：`core_pipeline.prompt_assets` — agent_name + prompt_stage 唯一活跃版本
- Genre：`core_pipeline.genre_adapters` — 按题材覆写（如"古装宫斗"）
- Instance：运行时 dict，不持久化
- 合成：`compose_prompt(master, genre_adapter, instance_vars)`
- 版本：`prompt_versions` 自动记录历史，`_bump_version` 递增
- 锁定：`locked_by` 字段防止自动修改

| 客户端 | 接口 | Mock | 真实实现 | 工厂 |

| MQ | `MQClient` | `MockMQClient` | `RocketMQClient` | `get_mq_client()` |

### 5.1 RAG 客户端
- 切换：`QDRANT_URL` 环境变量
- Qdrant 连接：`qdrant-client` SDK，`check_compatibility=False`
- Collection：`autoflow_rag`，dim=1536
- 操作：search（scroll+filter+sort）、upsert（deterministic point ID）、delete、count、health

### 5.2 MQ 客户端
- 切换：`ROCKETMQ_ENDPOINT` 环境变量
- 三级降级：native SDK (Linux) → gRPC proxy (macOS) → local buffer
- gRPC proxy：RocketMQ 5.x proxy 端口 8081
- Topic：autoflow.agent.task / agent.result / supervisor.alert / evolution.trigger
- 本地 buffer 管理：`get_buffer_stats()` / `flush_buffer()`

## 6. 成本守卫

- 硬约束：≤ 30 CNY/分钟成片 (`COST_BUDGET_PER_MIN`)
- 记录：`core_pipeline.cost_events`（cost_type: llm_call/gpu_render/audio_gen/storage/embedding）
- 三级状态：ok (< 70%) → warning (70-90%) → critical (> 90%)
- 检查点（Supervisor）：N02, N05, N09, N14, N17, N23
- 合规检查：`project_groups.compliance_rules` → `max_cost_per_episode`

## 7. Agent 注册表

- 文件：`backend/agents/registry.py`
- 模式：懒加载工厂（register class → instantiate on first `get_agent()`）
- 自动发现：`register_all_agents()` 注册 7 生产 + supervisor + evolution_engine
- API：`get_agent(name)` / `list_agents()` / `is_registered(name)`

## 8. SupervisorAgent

- 类型：纯规则引擎（无 LLM 调用）
- 职责：成本守卫 + 合规检查 + 预警广播
- 检查点：N02, N05, N09, N14, N17, N23
- 动作：ok → 日志; warning → MQ 广播; critical → MQ + cost_event + 标记人审

## 9. EvolutionEngine

- 4 模式：reflection / prompt_ab_test / lora_train(预留) / rag_cleanup
- reflection：LLM 分析近期 traces → 写入经验教训到 agent_memory
- prompt_ab_test：LLM 生成 Prompt 变体 → 记录到 evolution_runs
- rag_cleanup：删除低分/过期 RAG 案例
- 审计：`core_pipeline.evolution_runs` 记录每次进化执行

## 12. 与 LangGraph 的集成（编排运行时 Agent 负责）

主控 Agent 提供的框架层（BaseAgent + Registry + Memory + RAG + MQ）是"骨架"。
编排运行时 Agent 在 R6/R9/R12 任务中完成"肌肉"：

- R6：Worker handler 调用 `get_agent(name).execute(context)` 桥接 LangGraph 节点
- R9：挂载真实 LLM（通过 Prompt 资产 + OpenAI/Doubao API）
- R12：Supervisor 横切检查集成到 supervisor.py 路由层

框架层保证：任何生产 Agent 只要实现 `plan_episode()` + `execute_shot()` + `review_batch()`，即可被 Registry 管理、被 LangGraph 调度、自动记录 trace、自动成本追踪。非生产 Agent（Supervisor/EvolutionEngine）继续使用 `reason()` + `act()` 旧接口。
